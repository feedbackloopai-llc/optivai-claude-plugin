## The v1 integration gap → what VA0b builds

**Today:** worker → detached worktree → commit (orphaned) → worktree removed → V runs against stale working checkout → code-bead can't close.

**VA0b (worktree-preserving fix):**
1. `_live_worktree_manager` creates the worktree on a **named branch** `mayor/<bead_id>` (not detached) so commits are reachable.
2. The worker runs its work + commit **in the worktree**; **V runs with `cwd = worktree`** (sees the worker's changes in isolation).
3. The worktree lifecycle moves under the Mayor: on **V exit 0**, the **Mayor (single-writer)** merges `mayor/<bead_id>` → the working branch, **serialized** by a merge lock (mirrors GT's merge-slot); **then** closes the bead, **then** tears down the worktree. On **V≠0**: teardown without merge (code discarded), bead stays open.

This keeps worktrees AND makes V meaningful AND integrates code. It is a **minimal serial Refinery**; VB2 upgrades it to GT's batch-then-bisect.
