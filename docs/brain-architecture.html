Executive summary

What problem does this actually solve?

Vector RAG (what we had) treats memory as a search problem — find similar things. That's powerful but flat: it answers exactly one question. Six concrete things you couldn't do before and can now.

  1. 1

    "Where did this thought come from?"

    Before: a thought said "decided to go with Y because of X" — but you couldn't see the prior thought X that triggered it. Each capture was isolated. The brain was a bag of facts with no causal links.

    Now: every capture stamps wasDerivedFrom. /brain-trace <id> walks back to the original ancestor. You can see how a conclusion was built.

  2. 2

    "What did this look like before?"

    Before: if you updated a thought (or auto-capture rewrote one), the old version was gone. No history.

    Now: /brain-inspect <id> --at-revision 1 returns the original state. /brain-inspect <id> --at 2026-04-15T12:00:00Z returns whatever was there on April 15. Revisions never get rewritten — even after a rollback creates a new revision, the originals stay queryable forever.

  3. 3

    Actually forget something and prove it.

    Before: DELETE FROM brain.thoughts WHERE thought_id = X removes the row, but: are there embedding neighbors that still surface it? Did metadata cross-reference it? If someone audits this later, what's your evidence the data is gone?

    Now: /brain-forget runs 300 probes against the post-delete store, accepts only if k=0 surface, records dual-bound confidence (99.9999793% exact binomial) in an audit row. If any probe leaks, it restores from snapshot. Provable forget.

  4. 4

    Tell the brain "this thought matters more than its similarity score suggests."

    Before: ranking was purely cosine + keyword + recency. There was no way to say "this turned out to be important." The brain didn't learn from feedback.

    Now: /brain-promote adds a Hebbian weight (time-decayed (1 + days_since)^-0.7). Boost is gated by a within-kind relevance floor so a heavily-promoted-but-irrelevant thought can't outrank a relevant unpromoted one. The brain gets smarter about what to surface.

  5. 5

    Debug "why did the agent retrieve THAT thought before deciding?"

    Before: nothing. Black box.

    Now: brain.replay_log records every brain op with PII-redacted query + result + OTel trace correlation. /brain-replay --session-id X shows you the exact retrieval that triggered the agent's choice.

  6. 6

    Run this on a partner's machine without bleeding contexts.

    Before: $USER env var was the entire isolation story. Anyone could export USER=alice and read alice's brain.

    Now: Principal Scoping (PS) enforced at every read/write/forget/promote entry point. Cross-user attempts raise RuntimeError, not silent empty-list. Tested via 25+ multi-tenant isolation tests. (Still per-user-Neon at the deployment layer — but the substrate is multi-tenant-safe if you ever want to revisit that.)

The deeper difference

Vector RAG treats memory as a search problem: "find similar things." That's powerful but flat. What v0.2.0 adds is the ability to reason ABOUT the memory, not just retrieve FROM it. Each primitive maps to a cognitive function:

Primitive What it answers
PV (provenance)"Where did I learn this?"
RB (versioning)"What did I used to think?"
VF_ε (verified forget)"Stop bringing that up — and prove it's gone."
Hebbian (promote/demote)"This is what matters."
Replay log"How did I get here?"

Vector RAG answers exactly one question: "what's similar to this query?" That's still the foundation — every primitive sits ON TOP of pgvector cosine. We didn't replace the search engine; we added context to what the search engine retrieves and what happens after.

What you'll actually notice

Most days: your existing /brain-search works the same, just with Hebbian-boosted ranking when you've promoted something. /brain-capture works the same, just stamping provenance silently. The new behavior surfaces when:

  • You want to undo something you regret capturing → /brain-forget
  • The agent does something weird and you want to know what it retrieved → /brain-replay
  • A thought says "X but actually we decided Y later" → /brain-trace shows the chain; /brain-inspect shows what X was originally
  • You want to reinforce a pattern after the agent finally got it right → /brain-promote

What it's NOT

It's not a different brain. It's the same pgvector substrate with five new capabilities layered on. If you turned the new primitives off tomorrow, search would still work — you'd just lose the explainability + auditability + verified-forget guarantees.

The honest one-liner: vector RAG with operational rigor. Same intelligence, much more accountability.

The "before"

What v0.1.x couldn't do

v0.1.x was vector RAG with auto-extracted metadata — pgvector cosine search over thoughts captured by a Claude Haiku metadata extractor. It was effective for "find similar things." It was structurally lossy for everything else.

  • ·/·No way to ask where did this thought come from? — no provenance trail back to the originating agent, activity, or source URI.
  • ·/·No way to ask what did this thought look like last week? — no versioned history; updates overwrote the row.
  • ·/·No way to verifiably forget something — DELETE ran but nothing proved the embedding-space residue was gone.
  • ·/·No way to tell the brain this one matters more than its raw similarity score — no Hebbian boost lane.
  • ·/·No audit trail of brain operations — searches, captures, and forgets vanished into the ether.
MS_ε · five primitives

Mnemonic sovereignty, from the paper to the substrate

From Lin / Li / Chen 2026, Mnemonic Sovereignty in Language Model Memory Systems (arXiv 2604.16548), §12.1. The framework defines memory as sovereign iff five properties hold simultaneously — and they have a dependency pre-order, which is not aesthetic. It is the engineering build order.

Without provenance you can't rollback meaningfully. Without rollback you can't verify forgetting. The Wave-1 shipping order — PV, then RB, then VF_ε — was not a choice; it was the only order this could be built in.

Wave-2 · three magic moments

What you can do that v0.1.x couldn't

Three "wow" capabilities layered on the substrate. Each is one CLI flag, but each requires every primitive underneath to work.

CITATION --trace

Walk a thought's provenance chain back to the original ancestor. Handles cycles, max-depth, and orphans where parents were forgotten. Useful for "where did this knowledge come from?"

INSPECT --inspect

Query historical state by ISO timestamp or revision number. Because rollback never rewrites, revision 1 stays queryable forever — even after a rollback that produced revision N+M.

REPLAY-LOG --replay

Chronological PII-redacted OTel-correlated audit trail of every brain op. Useful for "what did the agent retrieve before deciding X?"

VF_ε · the forget guarantee

The 99.9999793% number, in math you can run

At n=300 probes, k=0 surfaced, ε=0.05 target: the residue probability is dual-bounded. The exact-binomial confidence is the headline number; the Hoeffding bound is recorded alongside for distribution-free conservatism. Both go into brain.forget_audit as distinct, labeled fields — so a reader can always see which number is being quoted.

99.99998%
exact-binomial
# Exact binomial · k=0 · n=300 · ε=0.05 (the headline) 
python3 -c "print((1 - 0.05) ** 300)"
# 2.075303347768056e-07
python3 -c "print(1 - (1 - 0.05) ** 300)"
# 0.9999997924696653
# Hoeffding one-sided · same params (the conservative complement) 
python3 -c "import math; print(math.exp(-2 * 300 * 0.05**2))"
# 0.22313016014842982   # residue bound
# → 77.69% confidence · distribution-free, loose

The audit row carries exactBinomialBound, exactBinomialConfidence, hoeffdingBound, hoeffdingConfidence, and probeQuality as distinct labeled fields. The 99.9999793% figure is the exact binomial confidence (5 sig figs), never the Hoeffding bound.

VF_ε · delete-after-verify

Why the naive ordering loses data, and the shipped pattern doesn't

The naive design — "verify first, then delete" — can't work: probes need the post-deletion state to be queryable, which means the delete has to happen first. The next-naive design — "delete, verify, rollback if leak" — had a data-loss window between delete and verify. The shipped pattern collapses both failure modes: snapshot the live state in memory, delete, probe against the post-delete store, and either accept or restore from the snapshot.

STEP · 01

Snapshot

Capture forgotten_text + top-50 semantic neighbors into in-memory ProbeSeedSnapshot. Live row untouched.

STEP · 02

Delete

Remove the live row. thought_versions cascade-deletes. Embedding space now reflects the post-delete reality.

STEP · 03

Probe

n=300 queries against the post-delete store, distributed 40 / 30 / 20 / 10 across semantic / paraphrase / partial-fragment / embedding-perturb.

STEP · 04

Accept or restore

If k=0: commit + audit forgotten. If k>0 or verify errors: restore from snapshot + audit forget-failed-residue.
